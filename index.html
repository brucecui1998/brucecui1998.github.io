使用命令行查看u盘

在linux系统下，我们可以通过使用命令行的方式来查看u盘。首先需要找到u盘的挂载点，通常情况下，u盘会自动挂载在‘/media’目录下。我们可以通过输入命令‘cd /media && ls’来进入该目录并查看当前目录下的文件和文件夹。

如果u盘没有自动挂载，我们可以手动挂载u盘。首先需要创建一个mount_point用来作为该u盘的挂载点。比如我们可以在‘/mnt’目录下创建一个名为‘usb’的文件夹来作为挂载点。然后，我们可以通过输入命令‘sudo mount /dev/sdb1 /mnt/usb’来挂载该u盘到‘/mnt/usb’目录下。

挂载u盘

最后，我们可以通过输入命令‘cd /mnt/usb && ls’来进入该目录并查看u盘中的文件和文件夹。

英伟达在MLPerf v1.0中U-Net3D的优化思路

U-Net3D is one of the two new workloads in this round of MLPerf training. We used the following optimizations:

  • Spatial parallelism
  • Asynchronous evaluation
  • Data Loader
  • Channels-Last layout support

英伟达在MLPerf v1.1中U-Net3D的优化思路

矢量化串联和拆分操作

3D-UNet 使用串联操作来连接解码器和编码器激活。这会导致在向前和向后传递中激活张量的设备到设备副本。我们通过使用矢量化加载和存储来优化这些副本，执行 4 倍宽的读/写操作。这使连接和拆分运算符的速度提高了 2.4 倍以上，在单节点配置上实现了 4.7% 的端到端加速，在最大规模配置上实现了 1.3% 的端到端加速。

高效的空间平行卷积

在 MLPerf v1.0 中，我们引入了空间并行卷积，我们将输入激活拆分到多个 GPU（准确地说是 8 个）。空间平行卷积的实现使我们能够将光晕交换隐藏在卷积后面的卷积后面。

在 MLPerf v1.1 中，我们优化了通信和卷积操作的调度，以便在启动的通信和卷积内核之间获得更好的重叠。虽然这确保了光晕交换不会暴露，但它也有助于显着减少抖动。这种优化的调度将最大规模配置中的分数提高了 25% 以上。

更好的配置

我们将全局批大小增加到数据集大小的一个因素。DALI 数据加载器库使我们能够使用相同的分片来训练不同的时期。这使我们能够显著减少在 GPU 中缓存数据集所需的时间。

由于每个 GPU 加载的图像要少得多，因此 DALI 中的边界框缓存预热速度也快得多。此优化显著缩短了启动时间，比 MLPerf v1.0 加速了 20%。

数据并行异步评估

随着训练速度越来越快，扩展评估以隐藏在训练后面变得具有挑战性。在 MLPerf v1.1 中，对单个图像的推理跨 GPU 进行分片，以提高评估缩放。然后将推理的结果全部收集起来，形成最终输出。这使得整个评估阶段隐藏在训练迭代后面。

更快的组实例规范

通过并行化多个通道块的 GPU 间通信，并通过矢量化内存读写来减少内核的 DRAM 时间，显著改进了多 GPU 实例范数内核。这导致最大规模配置的吞吐量提高了 5% 以上。

英伟达在MLPerf v2.0中U-Net3D的优化思路

3D U-Net具有多个大层，其输入通道计数为32。对于wgrad内核，使用默认64x256x64的内核意味着显著的瓦片大小量化损失。 由于在cuDNN中引入了新的32x256x32 wgrad内核，节省了这种瓦片大小的量化损失。这导致MLPerf v2.0中的单个节点比MLPerf v1.1的速度提高了5%以上。

英伟达在MLPerf v2.1中U-Net3D的优化思路

In MLPerf Inference v2.0, the 3D U-Net medical imaging workload switched to the KITS19 dataset, which increased image sizes by up to 8x and raised the amount of compute processing required for a given sample up to 18x due to the sliding window inference. For more information about the NVIDIA MLPerf Inference v2.0 submission, see Getting the Best Performance on MLPerf Inference 2.0.

For MLPerf Inference 2.1, we further improved the performance of the first convolution layer with the TensorRT IPluginV2DDynamicExt plugin.

KiTS19 images are single-channel tensors, and this challenges the performance of the very first 3D convolution in 3D U-Net. In 3D convolution, this channel dimension typically contributes to GEMM’s K dimension. This is specially relevant because the overall performance on 3D U-Net is dominated by the first two and last two 3D-Convolutions. In MLPerf Inference v2.0, these four convolutions contributed to roughly 38% of the entire network run-time; the very first layer responsible for 8%. A non-trivial factor that explains that is the need to use zero-padding to accommodate for the NC/32DHW32 vectorized format layout in where the tensor cores can be utilized most efficiently.

With our updated plugin, we use a INT8 Linear format to leverage efficient computation on this single-channel limited 3D shape input. The advantages of this are twofold:

  • Higher effective use of flops: by not performing unneeded computations
  • PCIe transfer B/W savings: avoids the overhead of either moving zero padded input tensor between host and GPU memory or zero-padding on GPU before sending the input tensor to TensorRT

This optimization improved the first layer performance by 2.7x. Additionally, the slicing kernel no longer needs to deal with zero-padding, and therefore its performance also improved by 2x. As a net result, 3D-UNet’s end-to-end performance improved by 5% in MLPerf Inference 2.1.

英伟达在MLPerf v3.0中U-Net3D的优化思路

NVIDIA 提交了 432 台 NVIDIA H100 Tensor Core GPU 的结果，创下了 0.82 分钟（49 秒）训练基准的新纪录。与上一轮相比，H100 的每加速器性能也提高了 8.2%。

为了大规模实现出色的性能，更快的 GroupBatchNorm 内核是一项关键优化。

在我们最大规模的 3D U-Net 提交中，神经网络中的实例归一化操作需要对四个 GPU 的张量均值和方差执行缩减。通过使用更快的 GroupBatchNorm 内核来实现实例规范化，我们实现了 1.5% 的性能提升。

1351. 统计有序矩阵中的负数

题目

给你一个 m * n 的矩阵 grid，矩阵中的元素无论是按行还是按列，都以非递增顺序排列。 请你统计并返回 grid 中 负数 的数目。

示例 1：

示例 2：

提示：

  • m == grid.length
  • n == grid[i].length
  • 1 <= m, n <= 100
  • -100 <= grid[i][j] <= 100

进阶：你可以设计一个时间复杂度为 O(n + m) 的解决方案吗？

思路

二分查找：

所以我们通过二分查找可以找到每一行中从前往后的第一个负数，那么这个位置之后到这一行的末尾里所有的数必然是负数了，可以直接统计

复杂度分析

时间复杂度：二分查找一行的时间复杂度为logm，需要遍历n行，所以总时间复杂度是O(nlogm)
空间复杂度：O(1)

倒序遍历：

考虑我们已经算出第 i 行的从前往后第一个负数的位置 posi，那么它下面的那个元素肯定也是负数。我们可以在遍历每一行时从后往前遍历，一旦遇到正数就break，遇到负数就记录它的位置，下一次接着从这个位置遍历

时间复杂度：考虑每次循环变量的起始位置是单调不降的，所以起始位置最多移动 每一行有多少列 次，时间复杂度 O(n+m)。
空间复杂度：O(1)
