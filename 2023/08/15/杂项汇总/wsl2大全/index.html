在wsl2中使用CUDA的局限有很多，例如

Linux CUDA 应用程序的已知限制 

下表列出了 WSL 2 上的已知限制，这些限制可能会影响使用 Linux 上完全支持的某些功能的 CUDA 应用程序。

局限性 冲击
不支持麦克斯韦 GPU。 Maxwell GPU 在 WSL 2 中不受官方支持，但它可能仍然可以工作。建议使用 Pascal 和更高版本的 GPU。
统一内存 - 完全托管内存支持在 Windows 本机上不可用，因此 WSL 2 在可预见的未来将不支持它。 UVM 完整功能将不可用，因此依赖 UVM 完整功能的应用程序可能无法工作。 如果应用程序使用的是托管内存，则应用程序可能会看到性能降低和系统内存使用率较高。 不支持并发 CPU/GPU 访问。CUDA 查询将说明它是否受支持，应用程序应检查这一点。
固定系统内存（例如：应用程序驻留用于 GPU 访问的系统内存）应用程序的可用性有限。 例如，某些深度学习训练工作负载（取决于所使用的框架、模型和数据集大小）可能会超过此限制，并且可能无法工作。
裸机（不是容器）上的根用户不会在预期位置找到 nvidia-smi。 用 /usr/lib/wsl/lib/nvidia-smi 或手动添加 /usr/lib/wsl/lib/ 到路径）。
使用 NVIDIA Container Toolkit for Docker 19.03，仅 --gpus all 受支持。 在多 GPU 系统上，无法使用特定索引号枚举 GPU 来筛选特定 GPU 设备。

尚不支持的功能 

下表列出了当前不支持的功能集。

局限性 冲击
NVML（nvidia-smi）尚不支持所有查询。 GPU 利用率、活动计算进程是一些尚不支持的查询。不支持可修改的状态功能（ECC、计算模式、持久性模式）。
OpenGL-CUDA Interop 尚不受支持。 依赖OpenGL的应用程序将无法工作。
